# 规范文档设计 说明

**不需要强行全用英语**，你的中文 Gherkin 文件已经足够好。以下是判断依据和建议：

---

### 一、当前写法完全可用
- Gherkin 关键字（`功能`、`场景`、`假如`、`当`、`那么`）已被主流框架支持（Cucumber/Behave 均提供中文关键词解析）。
- 步骤描述用中文更贴近领域专家（老师、产品经理），降低理解成本，符合 BDD 的“共同语言”原则。

---

### 二、唯一需统一的是**错误信息语言**
你目前的错误提示中英混杂（如 `"Group not found"` 与 `"错误信息"` 混合）。建议**二选一**：
- 若面向中文团队：全部改为中文错误信息（如 `"旅游团不存在"`）。
- 若为了代码中日志/异常统一：保留英文错误信息，但在 Feature 中保持英文提示（即把 `错误信息为` 改为 `错误信息为 "Group not found"` 不算错，但当前中英混合稍显突兀）。

---

### 三、英文使用
- 当团队中有非中文成员，或需要与国际工具链严格对齐时。
- 纯英文可使 Step 实现代码与 Feature 文字直接映射（如 `@when("the user pays deposit {amount}")` 比中文步骤更容易被正则匹配），但中文同样可通过正则实现。

---

### 四、建议
**保持现状，仅把错误信息统一为中文即可**。这样在整个演示场景中，所有内容（UI、文档、测试）都是中文，老师看起来更直观，也更能体现“业务人员可直接阅读可执行规范”的 SDD 理念。

如果你需要，我可以帮你把所有 Feature 文件中的英文错误提示批量替换为中文。
