# 旅游业务管理系统协议规约

> **版本**: 2.0  
> **适用阶段**: 系统设计、开发、测试、验收  
> **说明**: 本文档是旅游业务管理系统的业务协议标准，定义核心业务规则、流程、状态机及计算细节。具体接口定义由 `protocol.ts` 在后继对话中承载，本文档不包含端点路径、请求/响应体结构等实现细节。

---

## 1. 系统目标

为武汉XXX旅行社提供旅游业务申请的信息化管理，涵盖旅游团查询与申请、订金/尾款支付、参加者信息补全、催款单生成、财务导出以及取消/变更等全流程，确保数据一致性与业务合规。

---

## 2. 术语与概念

| 术语 | 定义 |
|------|------|
| **旅游路线 (Route)** | 旅行社设计的一条旅行产品线（如“武汉-神农架三日游”），可被启用或停用，停用后不再关联新的旅游团。路线变更需保留历史版本。 |
| **旅游团 (Group)** | 路线的具体发团批次，包含出发日期、报名截止日期、人数上限、大人/小孩价格，并可对外发布。 |
| **申请单 (Application)** | 旅客（申请责任人）发起的一次参团申请，关联一个旅游团，记录责任人信息、人数、费用和支付情况。一个申请单包含多位参加者（含责任人）。 |
| **参加者 (Participant)** | 申请单中的具体出行人员，其中一人标记为责任人（is_leader）。责任人也是参加者。 |
| **订金 (Deposit)** | 根据距出发天数按比例预先支付的款项。 |
| **尾款 (Balance)** | 总旅费扣除订金后剩余应付款项。 |
| **旅游确认书** | 申请信息齐全后生成的正式确认文件，包含团信息、参加者列表等。 |
| **余额交款单** | 通知旅客支付尾款的单据，含应付金额和最终支付期限。 |
| **催款员工** | 负责每日打印、邮寄确认书和交款单的员工。 |
| **财务系统** | 外部系统，通过本系统每晚导出的现金变动数据进行记账处理。 |

---

## 3. 核心流程概览

```
查询已发布旅游团 → 创建申请(状态 draft) → 支付订金(deposit_paid)
→ 录入参加者信息(info_complete=true) → 生成催款单(尾款未付时)
→ 支付尾款(confirmed) → [可变更参加者或取消申请]
```

任何非 `cancelled` 状态的申请均可被取消，取消时根据规则计算手续费并退款。

## 4. 实体关系及约束

### 4.1 路线 (Route)
- 属性：名称、描述、是否有效(`is_active`)、创建时间。
- 约束：停用的路线不可再用于创建新团。
- 变更历史：路线更新时应保留旧版本（通过自引用或版本表实现），不物理删除。

### 4.2 旅游团 (Group)
- 属性：路线ID、团代码(唯一)、出发日期、报名截止日期、最大人数(`max_pax`)、大人价格、小孩价格、是否发布(`is_published`)、创建时间。
- 约束：
  - 截止日期 < 出发日期。
  - 最大人数 > 0。
  - 价格一旦发布(`is_published = true`)即锁定，不可再修改价格字段。
- 发布前可修改团信息（含价格），发布后仅可查询/使用，不能变更价格。

### 4.3 申请单 (Application)
- 属性：团ID、责任人姓名/电话/邮箱/地址、大人数、小孩数、应付订金、应付总旅费、实付订金、实付尾款、状态、信息齐全标记(`info_completed`)、取消时间、创建时间。
- 人数约束：
  - 大人数 >= 1，小孩数 >= 0。
  - 总人数 = 大人数 + 小孩数 = 参加者数量。
- 金额约束：订金、总旅费在创建时由系统计算并固化，后续不因价格变动而改变。
- 状态枚举：`draft`, `deposit_paid`, `confirmed`, `cancelled`。

### 4.4 参加者 (Participant)
- 属性：申请单ID、姓名、性别、出生日期、电话、邮箱、地址、是否责任人、扩展信息(JSON)。
- 约束：
  - 每个申请至少有一名责任人。
  - 总人数必须等于申请单中的大人+小孩数（系统在录入参加者时校验）。
  - 责任人变更时，必须指定另一参加者为新责任人，禁止无责任人状态。

### 4.5 支付日志 (PaymentLog)
- 属性：申请单ID、支付类型(`deposit`/`balance`)、金额、创建时间。
- 用途：记录每一笔收入，保证财务导出准确。

### 4.6 退款记录 (Refund)
- 属性：申请单ID、扣除手续费金额、实际退款金额、退款时间。
- 仅当申请取消时生成。

### 4.7 财务导出记录 (FinancialExport，可选)
- 属性：导出日期、导出记录数、文件路径、创建时间。

## 5. 申请单状态机（协议核心）

```
                  ┌─────────┐
        创建申请  │  draft  │
       ─────────>│         │
                 └────┬────┘
         支付订金      │
                 ┌────▼────────┐
                 │ deposit_paid│
                 └────┬────────┘
       尾款结清        │
                 ┌────▼──────┐
                 │ confirmed │
                 └────┬──────┘
                      │
        取消(任意状态) │
                 ┌────▼──────┐
                 │ cancelled │
                 └───────────┘
```

状态说明：
- **draft**: 申请已创建，但订金未付。该状态不占用旅游团名额，可随时取消。
- **deposit_paid**: 订金已支付，名额已占用，可录入参加者信息。此时若 `info_completed = true` 且尾款未付，进入催款流程。可取消（按手续费规则退款）。
- **confirmed**: 尾款付清，申请完成。允许变更参加者信息或取消。
- **cancelled**: 申请已取消，终态，不可再进行任何操作。

`info_completed` 标记独立于状态，可在 `deposit_paid` 或 `confirmed` 阶段为真。其含义为“参加者信息已全部录入，且人数匹配”。仅当 `info_completed=true` 且状态为 `deposit_paid` 且尾款未付时，系统生成催款单。

## 6. 业务规则精要

### 6.1 订金计算

- **触发时机**: 创建申请时。
- **输入**: 服务器当前日期 `today`、旅游团出发日期 `departure`、大人数、小孩数、大人价格、小孩价格。
- **公式**:
  1. 总旅费 = 大人数 × 大人价格 + 小孩数 × 小孩价格
  2. 天数差 `days = (departure - today).days`
  3. 订金比例:
     - `days >= 60` → 10%
     - `30 <= days < 60` → 20%
     - `days < 30` → 100% (全款)
  4. 应付订金 = 总旅费 × 比例，金额四舍五入到分。
- **全款场景**: 若全款，应付订金 = 总旅费，后续无需支付尾款。

### 6.2 尾款支付期限

- **触发条件**: 信息齐全后，系统为该申请生成余额交款单时计算。
- **规则**:
  - 基础期限 = 出发日期 - 30 天
  - 保底期限 = 交款单生成日期 + 10 天
  - **最终支付期限 = max(基础期限, 保底期限)**
  - 若申请时已全款支付（订金即总旅费），则只生成确认书，无交款单，无期限。
- **注意**: 交款单生成日期指系统定时任务执行日期（催款单生成日期），不是申请创建日期。

### 6.3 取消与手续费

- **适用范围**: 任何非 `cancelled` 状态的申请。
- **手续费比例**（距出发天数 `D` = 出发日期 - 取消当天日期）：
  - `D >= 30` → 0%
  - `10 <= D < 30` → 20%
  - `1 <= D < 10` → 50%
  - `D = 0`（出发当天）→ 100%
  - 注意边界：当 `D` 恰好为 10 天时，适用 20%；恰好为 1 天时，适用 50%；0 天为出发当天。
- **计算步骤**:
  1. 已支付总额 `P = 实付订金 + 实付尾款`
  2. 手续费 `fee = P × 比例`
  3. 退款金额 = `P - fee`
  4. 生成退款记录，申请状态更新为 `cancelled`，取消时间记录当前时间。
- **并发控制**: 取消操作需在事务中执行，锁定申请行防止重复取消。

### 6.4 名额占用与并发控制

- **占用时机**: 支付订金时检查并占用名额。
- **名额计算**: 当前旅游团所有状态为 `deposit_paid` 或 `confirmed` 的申请单的总人数（大人+小孩）之和。
- **校验**: 支付订金时，`当前占用人数 + 本次申请人数 <= max_pax`。若超限，拒绝支付。
- **并发保证**: 使用数据库行级锁（如 `SELECT ... FOR UPDATE` 锁定旅游团行）确保检查与状态变更的原子性。

### 6.5 价格发布锁定

- 旅游团 `is_published` 初始为 `false`，此时可修改价格。
- 一旦执行“发布”操作，`is_published` 置为 `true`，系统前端和后端均禁止修改价格字段。
- 未发布的旅游团不在前台查询中展示。

### 6.6 催款单生成规则（定时任务）

- **执行频率**: 每日凌晨 1 点。
- **筛选条件**: 满足以下所有条件的申请：
  1. `state = 'deposit_paid'`
  2. `info_completed = true`
  3. `paid_balance = 0` (尾款未付)
  4. 最近一次信息更新日期为昨日（即前一天变更为信息齐全状态），通过 `info_completed_at` 或 `updated_at` 判断。
- **生成内容**:
  - **若已全款支付**（应付订金 == 总旅费）：仅生成旅游确认书。
  - **若未全款支付**：生成旅游确认书 + 余额交款单，交款单包含剩余应付金额（总旅费 - 已付订金）和最终支付期限（按 6.2 计算）。
- **输出形式**: 生成的数据通过 API 供催款员工查询、打印。

### 6.7 财务导出规则（定时任务）

- **执行频率**: 每日 23:59。
- **数据来源**: `payment_logs` 表。
- **导出内容**: 当日（`created_at` 为当天）所有支付日志记录，字段包括：支付时间、申请单ID、旅游团代码、支付类型(deposit/balance)、金额。
- **输出**: 导出为 CSV 文件，存储于指定目录，并在 `financial_exports` 表记录元数据。

### 6.8 变更业务细则

- **参加者信息变更**：
  - 仅在申请状态为 `confirmed` 时允许。
  - 可修改参加者的非关键字段（姓名、联系方式等）。
  - 若变更责任人：原责任人 `is_leader` 设为 `false`，目标参加者设为 `true`。变更后必须保证至少有一名责任人。
- **增加参加者**：不视为变更。若需增加，须创建新申请单（但原申请保持独立）。
- **取消整个申请**：按 6.3 规则计算退款，申请状态变更为 `cancelled`，生成退款记录。

## 7. 时间与货币精度约定

- **日期基准**: 所有天数计算均基于服务器当前日期 `date.today()`，忽略时区差异（统一使用服务器本地日期）。
- **金额精度**: 所有金额字段使用 `DECIMAL(10,2)` 类型，计算过程中保留两位小数，舍入模式为四舍五入。
- **天数边界**:
  - “距出发 X 天”指日期差，例如出发日期为 8 月 15 日，当前为 8 月 14 日，则 `days = 1`，适用 `1 <= days < 10` 规则。
  - 出发当天 `days = 0`，适用全款手续费。

---

## 8. 安全性及审计要求

- 所有涉及支付、取消的操作必须在事务中执行，保证原子性和数据一致性。
- 支付日志不可修改或删除，仅追加写入。
- 财务导出应记录操作时间与文件路径，便于追溯。
- 前端不得进行关键金额计算，所有计算以服务端返回为准。

## 9. 示例场景

### 9.1 标准流程示例

1. 员工查询到“武汉-桂林4日游”团（出发 2026-08-20，团费大人 2000，小孩 1000，已发布，有名额）。
2. 创建申请：责任人张三，2大人1小孩，总旅费 5000。当前日期 2026-06-05（距出发 76 天），订金比例 10%，应付订金 500。
3. 顾客支付订金 500，员工确认，状态变为 `deposit_paid`，名额占用 +3。
4. 参加者信息邮寄回，员工录入三人信息，责任人标记为张三，`info_completed` 变为 true。
5. 次日凌晨催款任务发现该申请（昨日刚信息齐全），生成确认书和交款单（尾款 4500，支付期限 = max(出发-30天=7月21日, 今日+10天=6月16日) = 7月21日）。
6. 顾客在 7 月 10 日支付尾款 4500，状态变为 `confirmed`。
7. 申请完成。

### 9.2 取消示例

- 接上例，若顾客在 8 月 1 日（距出发 19 天）要求取消，已支付总额 5000，适用 20% 手续费，手续费 1000，退款 4000，状态变 `cancelled`。
