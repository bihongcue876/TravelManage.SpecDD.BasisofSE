# 旅游业务管理系统协议规约

> **版本**: 2.0  
> **适用阶段**: 系统设计、开发、测试、验收  
> **说明**: 本协议定义旅游业务管理系统的业务规则、角色权限、数据概念、状态机及输出规范，不包含具体接口定义（接口细节由 `protocol.ts` 承载）。

---

## 1. 系统目标

为武汉XXX旅行社提供旅游业务申请的信息化闭环管理，涵盖：

- 旅游团查询与申请
- 订金/尾款支付（支持多种支付方式、分次支付、凭证上传）
- 参加者信息补全（支持批量导入、编辑历史、重复检查）
- 单据打印（订金收据、旅游申请书、确认书、交款单）
- 催款流程（自动生成催款单、邮件/短信提醒、交款单编号）
- 财务导出（多格式导出、对账报表、银行流水对比）
- 取消与变更（取消预览、部分取消、退款审批）
- 路线与旅游团批量导入
- **角色化界面**（登录认证、动态菜单、工作台面板）

## 2. 术语定义

| 术语 | 说明 |
|------|------|
| **旅游路线** | 旅行社设计的旅行产品线，可启用/停用，变更历史保留。 |
| **旅游团（活动）** | 某个出发批次，关联路线，包含出发日期、截止日期、人数上限、价格。 |
| **申请单** | 旅客发起的一次组团旅游申请，关联一个旅游团，记录责任人信息、人数、金额、支付状态。 |
| **参加者** | 申请单中的具体出行人员，其中一人标记为责任人（`is_leader`）。责任人也是参加者。 |
| **订金** | 根据距出发天数比例预先支付的款项。 |
| **尾款** | 总旅费减去已付订金后的剩余应付款。 |
| **旅游确认书** | 申请信息齐全后生成的确认文件，包含团信息和全部参加者。 |
| **余额交款单** | 通知旅客支付尾款的单据，含剩余金额和支付截止日期。 |
| **催款员工** | 负责每日打印、邮寄确认书和交款单的员工。 |
| **财务系统** | 外部系统，接收本系统每晚导出的现金变动数据进行记账。 |
| **角色** | 系统用户的职能分类，决定可访问的菜单和操作。 |

## 3. 核心流程概览

```
查询已发布团 → 创建申请(状态 draft) → 支付订金(deposit_paid)
→ 录入参加者(info_complete=true) → 生成催款单(尾款未付时)
→ 支付尾款(confirmed) → [可变更参加者或取消申请]
```

任何非 `cancelled` 状态的申请均可被取消，取消时根据规则计算手续费并退款。支持部分取消（按参加者）。

## 4. 实体关系与约束

### 4.1 路线 (Route)
- 属性：名称、描述、是否有效(`is_active`)、创建时间。
- 约束：停用的路线不可用于创建新团。
- 变更历史：路线更新时保留旧版本。

### 4.2 旅游团 (Group)
- 属性：路线ID、团代码(唯一)、出发日期、截止日期、最大人数(`max_pax`)、大人价格、小孩价格、是否发布(`is_published`)、创建时间。
- 约束：
  - `deadline < departure_date`
  - `max_pax > 0`
  - 价格一旦发布（`is_published = true`）即锁定，不可再修改价格字段。
- 发布前可修改所有信息（含价格），发布后仅查询，不可变价。

### 4.3 申请单 (Application)
- 属性：团ID、责任人姓名/电话/邮箱/地址、大人数、小孩数、应付订金、应付总旅费、实付订金、实付尾款、状态、信息齐全标记(`info_completed`)、取消时间、创建时间。
- 约束：
  - `adults >= 1`, `children >= 0`
  - 总人数 = 大人 + 小孩 = 参加者数量
  - 金额在创建时计算并固化，不受团价格后续变动影响。
- 状态枚举：`draft`, `deposit_paid`, `confirmed`, `cancelled`

### 4.4 参加者 (Participant)
- 属性：申请单ID、姓名、性别、出生日期、电话、邮箱、地址、是否责任人(`is_leader`)、扩展信息(`extra` JSON)、创建时间。
- 扩展字段示例：`id_number`（身份证号）、`special_requirements`（特殊需求）等。
- 约束：
  - 每个申请至少有一名责任人。
  - 总人数必须等于申请单中的 `adults + children`。
  - 变更责任人时必须指定新责任人，禁止无责任人状态。

### 4.5 支付日志 (PaymentLog)
- 属性：申请单ID、支付类型(`deposit`/`balance`)、金额、支付方式(`cash`/`bank_transfer`/`wechat`/`alipay`)、凭证路径、创建时间。
- 用途：记录每一笔收入，支撑财务导出。

### 4.6 支付凭证 (PaymentVoucher)
- 属性：支付日志ID、文件名、文件路径、文件大小、创建时间。
- 关联一次支付，支持多张凭证上传。

### 4.7 退款记录 (Refund)
- 属性：申请单ID、可选的参加者ID（部分取消时）、手续费、退款金额、原因、退款渠道、状态(`pending`/`approved`/`rejected`/`completed`)、审批人、审批时间、退款时间。
- 仅当取消时生成。金额≥5000元的退款需审批。

### 4.8 参加者编辑历史 (ParticipantEditHistory)
- 属性：参加者ID、字段名、旧值、新值、操作人、创建时间。
- 用于审计参加者信息变更。

### 4.9 催款记录 (ReminderLog)
- 属性：申请单ID、提醒类型(`email`/`sms`/`print`)、内容、发送时间、是否成功。
- 记录所有催款动作。

### 4.10 交款单编号 (PaymentOrder)
- 属性：申请单ID、单据编号(唯一)、单据类型、创建时间。
- 用于生成余额交款单的唯一编号。

### 4.11 财务导出记录 (FinancialExport)
- 属性：导出日期、文件路径、文件格式、记录数、创建时间。

### 4.12 银行对账 (BankReconciliation)
- 属性：导入日期、文件名、总记录数、匹配数、未匹配数、创建时间。
- 包含明细项 (`BankReconciliationItem`)，记录银行日期、金额、参考号、匹配的支付日志ID。

## 5. 申请单状态机

```
创建申请 → draft  ──支付订金──→ deposit_paid  ──尾款结清──→ confirmed
  │                │                │                │
  └──取消──────────┴──取消──────────┴──取消──────────┘
                     ↓
                cancelled（终态）
```

- **draft**: 订金未付，不占用名额，可取消。
- **deposit_paid**: 订金已付，名额占用。可录入参加者。若 `info_completed = true` 且尾款未付清，进入催款流程。可取消。
- **confirmed**: 尾款付清，申请完成。允许变更参加者信息或取消（包括部分取消）。
- **cancelled**: 终态，不可再操作。

`info_completed` 是独立标记，可在 `deposit_paid` 或 `confirmed` 阶段为真。

## 6. 核心业务规则

### 6.1 订金计算
- 触发时机：创建申请时（或预览时）。
- 公式：
  1. 总旅费 = 大人数 × 大人价格 + 小孩数 × 小孩价格
  2. 天数差 `D = (出发日期 - 服务器当前日期).days`
  3. 订金比例：
     - `D >= 60` → 10%
     - `30 <= D < 60` → 20%
     - `D < 30` → 100% (全款)
  4. 应付订金 = 总旅费 × 比例，四舍五入至分。
- 全款场景：订金即总旅费，后续无尾款。

### 6.2 尾款支付期限
- 触发条件：信息齐全后生成余额交款单时计算。
- 规则：
  - 基础期限 = 出发日期 - 30 天
  - 保底期限 = 交款单生成日期 + 10 天
  - **最终支付期限 = max(基础期限, 保底期限)**
- 若全款支付，仅生成确认书，无交款单。

### 6.3 取消与手续费
- 适用范围：任何非 `cancelled` 状态的申请（包含部分取消）。
- 手续费比例（距出发天数 `D` = 出发日期 - 取消当天日期）：
  - `D >= 30` → 0%
  - `10 < D < 30` → 20%
  - `1 <= D <= 10` → 50%
  - `D = 0` → 100%（出发当天）
- 计算步骤：
  1. 已支付总额 P = 已付订金 + 已付尾款
  2. 手续费 = P × 比例
  3. 退款金额 = P - 手续费
  4. 生成退款记录，退款金额 ≥ 5000 元时状态为 `pending` 等待审批；否则 `completed`。
- 部分取消：按取消人数比例计算已付金额，同等手续费比例计算退款。

### 6.4 名额占用与并发控制
- 占用时机：支付订金时。
- 当前占用人数 = 该团所有状态为 `deposit_paid` 或 `confirmed` 的申请单的人数总和。
- 支付订金时检查：`当前占用人数 + 本次申请人数 <= max_pax`，否则拒绝。
- 并发保证：使用数据库行级锁（`FOR UPDATE`）锁定旅游团行。

### 6.5 价格发布锁定
- `is_published` 初始为 `false`，可修改价格。
- 执行“发布”操作后置为 `true`，前后端均禁止修改价格。
- 未发布团不在前台查询结果中显示。

### 6.6 催款单生成（定时任务）
- 执行时间：每日凌晨 1:00
- 筛选条件：`state = deposit_paid`, `info_completed = true`, `paid_balance < total_price - paid_deposit`
- 生成内容：
  - 旅游确认书（所有符合条件申请）
  - 余额交款单（尾款未付清的申请），包含支付期限（按 6.2 计算）及交款单编号（自动生成）。
- 也支持手动触发单条催款（短信/邮件）。

### 6.7 财务导出（定时任务）
- 执行时间：每日 23:59
- 导出数据：当日所有支付日志记录，支持 CSV/Excel/JSON 格式。
- 记录存入 `financial_exports` 表。

### 6.8 变更业务细则
- **参加者信息变更**：仅在 `confirmed` 状态允许，可修改字段，记录编辑历史。
- **责任人变更**：必须选定另一参加者为新责任人，原责任人取消标记。
- **增加参加者**：不视为变更，需新建申请单。
- **取消申请**：整体取消或部分取消（`/partial-cancel`），按 6.3 计算退款。

### 6.9 重复参加者检查
- 录入参加者时，可按手机号或身份证号（在 `extra` 中）检查同团是否已存在，返回警告。

### 6.10 批量导入
- **路线导入**：Excel 模板，支持批量创建路线。
- **旅游团导入**：Excel 模板，校验路线存在、日期合法性等。
- **参加者导入**：Excel 模板，支持批量录入参加者（需校验人数匹配）。

## 7. 打印输出规范

### 7.1 订金收据
- 内容：申请单ID、申请人姓名、旅游团代码、出发日期、大人/小孩数、订金金额、支付日期、收款方名称。
- 数据由后端提供，前端渲染打印。

### 7.2 旅游申请书
- 格式：对应表2，包含旅游团代码、出发日期、路线名，全部参加者信息（姓名、性别、出生日期、电话、地址、身份证号、特殊需求等），并标明责任人。
- 支持批量打印（多申请合并打印）。

### 7.3 旅游确认书 / 余额交款单
- 确认书：包含申请基本信息、参加者列表。
- 交款单：包含剩余应付金额、支付截止日期、交款单编号。
- 支持批量生成并打印。

## 8. 角色与权限

### 8.1 角色定义
| 角色标识 | 名称 | 职责范围 |
|----------|------|----------|
| `frontdesk` | 前台 | 查询旅游团、创建申请、支付订金/尾款、录入参加者、取消申请、打印收据/申请书 |
| `collector` | 催款员 | 查看催款清单、标记已打印、查看申请（只读）、发送催款提醒 |
| `product_admin` | 产品管理员 | 管理路线、管理旅游团（创建、修改、发布、导入）、查看所有申请数据 |
| `finance_admin` | 财务/管理员 | 查看财务流水、导出财务数据、管理用户、审批退款、查看所有模块 |

### 8.2 菜单权限原则
- 登录后，后端返回当前用户可访问的菜单项，前端动态渲染。
- 同一页面内，根据角色控制按钮显示（如 `AppDetail` 页面前台可见操作按钮，催款员仅可见“标记已打印”）。

### 8.3 工作台面板
登录后首页展示与角色相关的统计卡片和快捷入口：
- **前台工作台**：今日新增申请数、待录入参加者数、快捷创建申请。
- **催款工作台**：今日应催款数量、逾期未付数量、快捷进入催款清单。
- **产品工作台**：近期（如30天内）出发团数量、未发布价格团数量、快捷管理路线/团。
- **财务工作台**：昨日现金流水总额、昨日导出记录数、退款待审批数量、快捷进入报表。

## 9. 安全与审计

- 用户认证：JWT Token 方式，登录成功后返回 `access_token` 和角色。
- 所有请求需携带 Token，后端校验权限。
- 关键操作（支付、取消、审批退款）在数据库事务中执行。
- 支付日志不可修改或删除，仅追加。
- 财务导出记录和催款日志长期保留。
